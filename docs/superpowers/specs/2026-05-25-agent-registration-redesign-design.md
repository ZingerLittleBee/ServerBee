# Agent Registration Redesign

**Status:** Draft (rev 4)
**Author:** ZingerLittleBee
**Date:** 2026-05-25

## Problem

The current agent onboarding flow has three related problems:

1. **"Add Server" doesn't create a server.** Clicking *Add Server* only mints a free-floating enrollment code. The actual `servers` row is created by `POST /api/agent/register` **after** an agent successfully connects. There is no UI representation of "a server I plan to install" — it appears in the list only once the agent comes online.

2. **"Add Server" silently restores existing servers.** The register handler does fingerprint-based dedup: if the new agent's `fingerprint` matches an existing server, it reuses that `server_id` and rotates its token. From the operator's perspective, clicking *Add Server* sometimes creates a brand-new entry and sometimes silently restores an old one.

3. **Recover Agent is over-engineered.** The existing `recovery-merge` flow (recovery_job state machine, candidate scoring, rebind WebSocket coordination, recovery_lock service) only works when there is *another online agent* to graft the offline record onto. The normal "I reinstalled this VPS, give me a fresh install command" path is not directly supported.

## Goals

- *Add Server* always creates a new `servers` row, immediately, with all operator-supplied metadata (name, group, tags, billing, remark). The row exists in a `pending` state until an agent connects.
- The enrollment code generated by *Add Server* is bound 1:1 to that pre-created `servers` row. An agent using that code can only bind to that specific row.
- *Recover Agent* opens a dialog that produces a new install command bound to an existing server row. Available on both `online` and `offline` servers (not on `pending` — those already have an outstanding code or use *Regenerate code* on the card).
- Fingerprint-based silent restoration is removed entirely. The enrollment code is the sole authority for which server an agent binds to.
- The complex `recovery-merge` apparatus (state machine, candidate scoring, online-rebind, freeze guards) is removed in full.

## Non-Goals

- WebSocket-layer authentication (token + Argon2 verification), agent local capability probing, and the `effective_caps = server_caps & agent_local_caps` rule are unchanged.
- `--caps` selection in the dialog continues to be encoded into the `install.sh` command line, not persisted onto the server row. Post-install capability editing remains the job of the existing `capabilities-dialog`.
- iOS / mobile clients are out of scope. They consume `/api/servers` and gain `pending` as a new derivable status; they do not initiate registration.
- **Backward-compatible data migration is out of scope.** The project is in alpha. The migration is allowed to be destructive: legacy enrollments and recovery_job are wiped; the `servers` table is preserved but its token columns become nullable.

## Architecture

### Status model

A `servers` row is in exactly one of three derived states (no status column is stored):

| State | Definition | When it appears |
|---|---|---|
| `pending` | `token_hash IS NULL` | After *Add Server*. Also after *Recover* with `revoke_immediately = true`. |
| `online` | `token_hash IS NOT NULL` AND `agent_manager` has a live connection | Normal operation. |
| `offline` | `token_hash IS NOT NULL` AND `agent_manager` has no connection | Agent disconnected; or *Recover* with `revoke_immediately = false` where the old agent is still connected (in that subcase status is actually `online` until the new agent takes over). |

#### Browser representation

The frontend derives status from two inputs:

1. The server DTO returned by `/api/servers` and pushed via `BrowserMessage::FullSync` / `Update` gains a new `has_token: bool` field (= `token_hash IS NOT NULL`). The raw hash and prefix are never sent.
2. The existing `BrowserMessage::ServerOnline { server_id }` / `ServerOffline { server_id }` messages continue to indicate live connection state.

Status = `pending` if `!has_token`; otherwise `online` if the latest connection event was `ServerOnline`, else `offline`. No new `BrowserMessage` variant is required.

When *Add Server* creates a row, the server is broadcast via the existing per-server `Update` event with `has_token: false`. When *Recover* with `revoke_immediately = true` clears the token, the same `Update` mechanism re-broadcasts the row, plus a `ServerOffline` event drops any stale live-connection indicator.

### Enrollment lifecycle

`agent_enrollments` gains a binding column plus an explicit `revoked_at` column. Two terms are used throughout this spec — keep them distinct:

```
outstanding = consumed_at IS NULL AND revoked_at IS NULL
              (the row still occupies the per-server slot; may be expired)
usable      = outstanding AND expires_at > now()
              (an agent could currently redeem it)
```

The partial unique index covers `outstanding` (predicate must be deterministic, so no time clause). An expired-but-not-revoked row is still *outstanding* and still occupies the slot until something revokes it (the delete endpoint, or *regenerate-code* atomically superseding it). *Recover* never supersedes; it returns `409` when an outstanding row is present.

```sql
CREATE UNIQUE INDEX idx_enrollments_active_per_server
  ON agent_enrollments(target_server_id)
  WHERE consumed_at IS NULL AND revoked_at IS NULL;
```

State transitions:

- *mint* (Add Server / Recover / Regenerate) → row inserted with `consumed_at = NULL`, `revoked_at = NULL`, `expires_at = now + 10 min`
- *consume* (agent registers successfully) → `consumed_at = now`
- *revoke* (operator deletes code, or *Regenerate* supersedes an expired-but-not-revoked code) → `revoked_at = now`

Regenerate flow (operator clicks Regenerate on a pending card):

```
BEGIN
  -- optimistic concurrency: if caller passed expected_enrollment_id, verify it
  -- still matches the current outstanding row; abort with 409 if not.
  UPDATE agent_enrollments SET revoked_at = now()
    WHERE target_server_id = $1 AND consumed_at IS NULL AND revoked_at IS NULL;
  INSERT INTO agent_enrollments (target_server_id, ..., expires_at = now + 10m);
COMMIT
```

This makes the partial unique index hold for both regenerate-after-expiry and regenerate-while-still-usable. The `expected_enrollment_id` guard prevents the surprise where operator A sees a fresh plaintext code and operator B's concurrent regenerate silently invalidates it.

### Fingerprint as informational

`fingerprint` on `servers` is preserved as an informational column (so operators can see what hardware identity an agent reported), but the **unique index from `m20260329_000013_add_server_fingerprint.rs` is dropped**. With server-bound enrollments, multiple servers can validly share a fingerprint (e.g., two VMs cloned from the same image, two pre-created rows for the same host, etc.). The register handler writes whatever fingerprint the agent reports and does no lookup with it.

### Token revoke semantics in Recover Agent

The Recover dialog has a `Revoke existing token immediately` checkbox (default checked).

- **Checked**: Within the same transaction that mints the enrollment, `servers.token_hash` and `token_prefix` are set to `NULL`. After commit, `agent_manager.remove_connection(server_id)` is called. The old agent is kicked and cannot reconnect (any attempt fails `Unauthorized`). Server enters `pending` until the new agent's register call sets a fresh token.
- **Unchecked**: Only the enrollment is created. The old token stays valid and the old agent stays online. When a new agent calls `POST /api/agent/register` with the new code, the register handler (transactionally) overwrites `token_hash`/`token_prefix` — but does **not** close the old WS. The displacement happens later, when the new agent opens its WebSocket to `/agent/ws` and `AgentManager::add_connection` (`crates/server/src/service/agent_manager.rs:142`, see also `crates/server/src/router/ws/agent.rs:356`) overwrites the current connection slot for that `server_id` via `server_lifecycle_locks`. There is therefore a brief window — from "register call returns 200" until "new WS connects" — during which the old socket is still authoritative and can keep writing metrics. The window closes deterministically: the new agent's reporter opens its WS within seconds of `register` returning. Recording this explicitly so future maintenance does not mistake it for a leak or assume the register handler force-closes the old socket.

## Components

### Backend

#### Data model migration

A single migration captures all schema changes. Since the project is alpha and the user authorized destructive migration, no data is preserved beyond the `servers` table.

```rust
// crates/server/src/migration/m{N}_agent_registration_redesign.rs
// up():
//   1. Drop fingerprint unique index
DROP INDEX IF EXISTS idx_servers_fingerprint;

//   2. Make servers token columns nullable (SQLite: table rebuild)
ALTER TABLE servers ALTER COLUMN token_hash   DROP NOT NULL;
ALTER TABLE servers ALTER COLUMN token_prefix DROP NOT NULL;

//   3. Drop legacy enrollment + recovery tables and recreate clean
//      (existing table is singular `recovery_job`, see
//       m20260416_000017_create_recovery_job.rs)
DROP TABLE IF EXISTS agent_enrollments;
DROP TABLE IF EXISTS recovery_job;

CREATE TABLE agent_enrollments (
  id                TEXT PRIMARY KEY NOT NULL,
  code_hash         TEXT NOT NULL,
  code_prefix       TEXT NOT NULL,
  target_server_id  TEXT NOT NULL REFERENCES servers(id) ON DELETE CASCADE,
  created_by        TEXT NOT NULL REFERENCES users(id),
  expires_at        TIMESTAMP NOT NULL,
  consumed_at       TIMESTAMP,
  revoked_at        TIMESTAMP,
  created_at        TIMESTAMP NOT NULL
);
CREATE UNIQUE INDEX idx_enrollments_active_per_server
  ON agent_enrollments(target_server_id)
  WHERE consumed_at IS NULL AND revoked_at IS NULL;
CREATE INDEX idx_enrollments_code_prefix ON agent_enrollments(code_prefix);

// down() is a no-op per project rules.
```

Notes:
- The `label` column from the old enrollment schema is dropped (the operator-friendly name lives on the server row).
- No `revoke_on_mint` column is added (was in earlier draft, removed: the choice is consumed at request time and doesn't need to be persisted).

#### API surface changes

| Endpoint | Change |
|---|---|
| `POST /api/agent/register` | No longer creates `servers` rows. Calls `EnrollmentService::verify_and_consume_tx(tx, bearer)` whose contract is **explicitly** `consumed_at IS NULL AND revoked_at IS NULL AND expires_at > now()` (i.e. usable). A revoked-but-not-consumed code returns 401 here just like an expired one. Reads `target_server_id` from the returned row → sets that server's `token_hash` / `token_prefix` / `last_remote_addr` / `fingerprint`. All steps run inside a single sea-orm `TransactionTrait::transaction` so a failure to update the server rolls back the enrollment consume. Fingerprint is recorded for informational use only. |
| `POST /api/servers` *(new)* | Admin-only. `max_servers` cap is **soft**: a non-locking `COUNT(*) ≥ cap` check runs before the transaction; two concurrent requests can both pass the count and exceed the cap by 1–2 rows. This matches today's behavior in the existing register handler and is acceptable for the operator-driven path. In one transaction: insert pending server (token_hash NULL), insert server_tag rows for chosen tags, insert default network probe targets (existing `NetworkProbeService::apply_defaults` logic moves here), insert bound enrollment. Response: `{ server_id, enrollment: { id, code, code_prefix, expires_at } }`. The enrollment `id` is included so the frontend can pass it as `expected_enrollment_id` to later regenerate calls and as the resource id for DELETE. |
| `POST /api/servers/{id}/recover` *(new)* | Admin-only. Body: `{ revoke_immediately: bool }`. Rejects with `400` if the server is `pending`; rejects with `409` if an outstanding enrollment already exists for this server (operator must explicitly revoke it first — recover never silently supersedes). In one transaction: if `revoke_immediately`, NULL out token columns; insert new bound enrollment. After commit: if `revoke_immediately`, call `agent_manager.remove_connection(id)`. Response: `{ enrollment: { id, code, code_prefix, expires_at } }`. |
| `POST /api/servers/{id}/regenerate-code` *(new)* | Admin-only. Used on a `pending` server. Optional body: `{ expected_enrollment_id?: string }`. If provided and does not match the current outstanding enrollment's id (including the case where the operator's stale view expected a row but none exists now), returns `409`. In one transaction: revoke any outstanding enrollment (sets `revoked_at`), insert new bound enrollment. Server stays pending. Response: `{ enrollment: { id, code, code_prefix, expires_at } }`. This is the only path that auto-supersedes; *recover* does not, by design. |
| `POST /api/agent/enrollments` | **Removed.** |
| `GET /api/agent/enrollments` | Kept; returns `{ id, target_server_id, code_prefix, expires_at, consumed_at, revoked_at, created_at }`. `id` is required by the UI for `DELETE /api/agent/enrollments/{id}` and for `expected_enrollment_id` on regenerate calls. Plaintext code is never returned. |
| `DELETE /api/agent/enrollments/{id}` | Kept. Marks the enrollment `revoked_at = now()`. **Does not delete the server**, even if the server is pending. |
| `DELETE /api/servers/{id}` | Behavior extended (bundled with this redesign): after the cascading row delete, call `agent_manager.remove_connection(id)` to drop any live WS connection. Today's handler at `crates/server/src/router/api/server.rs:472` / `crates/server/src/service/server.rs:256` does not do this, which leaves the agent looping on reconnect. The cascade through `agent_enrollments` (via FK) is automatic. This is the only path that removes a server row. |
| `POST /api/agent/{id}/rotate-token` | Kept, but adds a guard: returns `400 BadRequest` if `token_hash IS NULL` (the server is pending; rotation is meaningless without a current token). For non-pending servers, behavior is unchanged. |
| `/api/server-recovery/*` (all variants) | **Removed.** |

#### Register handler simplification

```rust
async fn register(state, addr, headers, body) -> RegisterResponse {
    check_register_rate(&ip)?;
    let bearer = parse_bearer(&headers)?;
    let body_fp = body.as_ref()
        .and_then(|b| Some(b.fingerprint.clone()).filter(|f| !f.is_empty()));
    if let Some(ref fp) = body_fp { validate_fingerprint_format(fp)?; }

    let (server_id, plaintext_token) = state.db.transaction(|tx| async move {
        // verify_and_consume_tx contract:
        //   - Argon2-verifies bearer against rows where code_prefix = bearer[..8]
        //   - Only accepts rows where consumed_at IS NULL AND revoked_at IS NULL
        //                            AND expires_at > now()
        //   - On match: sets consumed_at = now() in the same tx; returns the row
        //   - On no match (wrong/expired/revoked/already-consumed code): returns None
        let enrollment = EnrollmentService::verify_and_consume_tx(tx, &bearer).await?
            .ok_or(AppError::Unauthorized)?;

        let server = server::Entity::find_by_id(&enrollment.target_server_id)
            .one(tx).await?
            .ok_or_else(|| AppError::Internal("Bound server vanished".into()))?;

        let plaintext = AuthService::generate_session_token();
        let hash      = AuthService::hash_password(&plaintext)?;
        let prefix    = plaintext[..8.min(plaintext.len())].to_string();

        let mut active: server::ActiveModel = server.clone().into();
        active.token_hash      = Set(Some(hash));
        active.token_prefix    = Set(Some(prefix));
        active.last_remote_addr = Set(Some(ip.clone()));
        active.fingerprint     = Set(body_fp.clone());
        active.updated_at      = Set(Utc::now());
        active.update(tx).await?;

        Ok::<_, AppError>((server.id, plaintext))
    }).await?;

    let _ = AuditService::log(&state.db, "system", "agent_enrolled",
        Some(&format!("server_id={server_id} prefix={}", &plaintext_token[..8])), &ip).await;
    // Broadcast the row change (has_token: true) so the UI flips out of pending.
    // Do NOT broadcast ServerOnline here — that event is owned by AgentManager::add_connection
    // and only fires once the WS connection is actually established. Otherwise a successful
    // register + failed WS would show the server as online when no agent is connected.
    broadcast_server_update(&state, &server_id);

    ok(RegisterResponse { server_id, token: plaintext_token })
}
```

The transaction is the critical fix: a panic / DB error after `verify_and_consume` would otherwise burn the enrollment code while leaving the server tokenless and unrecoverable.

#### Pending server card

Server cards with derived status `pending` render with:

- A new amber status indicator (distinct from green-online and gray-offline).
- "Waiting for agent…" placeholder in place of metrics.
- Enrollment summary line:
  - If a *usable* outstanding enrollment exists: `Code <prefix>… · expires in 7m 12s`
  - If outstanding but expired: `Code <prefix>… expired`
  - If no outstanding enrollment (revoked or never minted): `No outstanding enrollment code`
- **The plaintext install command is NOT available on the card.** Plaintext only exists in the `POST /api/servers` (or recover/regenerate) response. The card surfaces only the prefix and lifecycle status.
- Action menu:
  - **Regenerate code** — calls `POST /api/servers/{id}/regenerate-code`, switches the card or a small inline panel into "fresh code shown once" mode displaying the new plaintext code + install command.
  - **Delete server** — calls `DELETE /api/servers/{id}`, cascades enrollments.

This is the single most operator-visible consequence of the redesign: an enrollment code is a secret that can only be shown once; the UI must not pretend otherwise.

#### Recover Agent dialog

A new component `recover-agent-dialog.tsx`, opened from a server card's action menu when the server is `online` or `offline` (not `pending` — those use *Regenerate code*).

Layout:

- Server name: read-only text
- Caps: checkbox grid, editable (initial value = current `server.capabilities` projected back to keys)
- `Revoke existing token immediately` checkbox, default **checked**
  - When checked, a small warning shows: "the current agent will be kicked offline and cannot reconnect until the new install command runs"
- TTL tip: "Code valid for 10 minutes"
- Submit button: "Generate install command"
- On submit: `POST /api/servers/{id}/recover` → switches to install-command view (same layout as Add Server's post-submit state)

If the server has an outstanding enrollment when the operator opens Recover (race or stale page), the dialog shows an inline notice "An outstanding code exists (prefix `xxxx…`, expires in N min). Revoke it before generating a new one." with a "Revoke" button calling `DELETE /api/agent/enrollments/{id}`. Submitting the form in that state is blocked client-side, and the server-side `POST /api/servers/{id}/recover` would return `409` anyway. This is the deliberate "recover never silently supersedes" rule — autosupersession is only available via *Regenerate code* on pending cards.

## Errors & Edge Cases

| Case | Behavior |
|---|---|
| Outstanding enrollment exists when operator clicks Recover | Dialog shows the inline "outstanding code exists" notice with a Revoke button; submit is blocked client-side and `POST /api/servers/{id}/recover` returns `409` server-side. |
| Outstanding enrollment expires before agent connects (pending card) | Card status line shows "Code expired"; *Regenerate code* atomically revokes the old row and mints a fresh one. |
| Race: two operators click Regenerate simultaneously without `expected_enrollment_id` | Last-writer-wins. Both transactions serialize on the row's locks: A revokes the outstanding row + inserts its new code; B then revokes A's just-inserted code + inserts its own. Operator A's plaintext is silently invalidated. To avoid this, the UI always submits `expected_enrollment_id` set to whatever outstanding row id it observed when the dialog opened (or `null` for "no outstanding"); a mismatch returns `409` so the operator can reload and re-decide. |
| Race: Recover called twice on the same server | The first call inserts the outstanding row; the second sees it and returns `409` (per the recover-never-supersedes rule). No retry. |
| Operator deletes a pending server | `DELETE /api/servers/{id}` cascade-removes enrollments. No live connection to clean up. |
| Operator deletes an online/offline server | `DELETE /api/servers/{id}` is extended in this redesign (see API table) to call `agent_manager.remove_connection(id)` after the cascading row delete. Today's handler does **not** close the WS — this is a deliberate fix bundled with the redesign so the stale server-side connection is dropped immediately. The agent process itself keeps running its reconnect loop with the cached token (`crates/agent/src/reporter.rs:75`, `:1414`); subsequent reconnect attempts will fail with `Unauthorized` (the `servers` row is gone), and the agent's existing re-enrollment retry path is no help (the original enrollment code is consumed or also gone). Operator must stop the agent process on the VPS separately if they want the noise to cease. |
| Agent reaches register with valid code but bound server was deleted | Cascade prevents this in practice. If it does occur (race), respond `500 Internal`. |
| `max_servers` cap reached | `POST /api/servers` returns `400 BadRequest` before opening the transaction. |
| Two operators race on Add Server | Each transaction inserts its own server row; no conflict. |
| Agent restarts with cached token after `revoke_immediately = true` | Reconnect fails `Unauthorized`. Existing register-retry path is no help (the old enrollment is consumed). Operator must re-run install.sh. This is exactly the intent. |
| Rotate-token called on a pending server | `400 BadRequest` — pending servers have no token to rotate; the operator should use Recover instead. |
| Register handler partial failure (post-consume, pre-update) | The whole register call is one transaction; failure rolls back enrollment consume too, so the operator can retry with the same code. |
| Two cloned VMs share a fingerprint | Both register successfully (no unique index). The `fingerprint` column on each server records its self-reported value. |

## Security Considerations

- Enrollment codes remain Argon2-hashed with prefix lookup; binding to a `target_server_id` does not weaken the code itself.
- The partial unique index prevents stockpiling outstanding codes per server. Revoke + regenerate is the explicit superseding pattern.
- Removing fingerprint dedup eliminates a class of confusion (silent restoration). The trade-off — operator must consciously use Recover to reinstall an existing host — is intentional and aligns with the "no implicit state changes" principle.
- `revoke_immediately = true` is the Recover dialog's default. Operators must consciously opt out to keep an old agent running while a new code is outstanding.
- Plaintext enrollment codes are returned exactly once (at mint time) and never persisted to logs or audit metadata (only the prefix appears in audit lines).

## Testing

### Unit / integration tests added

- `POST /api/servers` creates server + tags + probe targets + enrollment in a single transaction; rolls back fully if any step fails.
- `POST /api/servers` respects `max_servers` as a soft cap: a single request beyond the cap is rejected; the test does not attempt to prove the cap is race-proof under concurrent inserts (it isn't, by design).
- `EnrollmentService::verify_and_consume_tx` rejects a revoked-but-not-consumed code with `None` (caller returns 401). Test must assert the server's token columns are NOT mutated when a revoked code is replayed.
- `EnrollmentService::verify_and_consume_tx` rejects an expired code with `None`.
- `POST /api/agent/register` with a bound enrollment updates the bound server's token columns and does **not** create a new server row.
- `POST /api/agent/register` ignores the `fingerprint` field for lookup (records but does not dedup).
- `POST /api/agent/register` rolls back enrollment consume if the server update fails (simulate by deleting the server row mid-transaction via a test hook, or by injecting a DB error).
- Partial unique index holds: while an outstanding enrollment exists, a direct second insert violates the constraint.
- `POST /api/servers/{id}/recover` with an outstanding enrollment returns 409 (no auto-supersession).
- `POST /api/servers/{id}/regenerate-code` with stale `expected_enrollment_id` returns 409; without it, last-writer-wins (covered by a documented test case).
- `DELETE /api/servers/{id}` on an online server closes the WS connection (verify `agent_manager.remove_connection` is invoked).
- `POST /api/servers/{id}/regenerate-code` revokes the prior outstanding enrollment and inserts a new one in one transaction; `expected_enrollment_id` mismatch returns 409.
- `revoke_immediately = true`: token columns become NULL, `agent_manager.remove_connection` is called, server enters pending state, an `Update` carries `has_token: false`, a `ServerOffline` event is fired.
- `revoke_immediately = false`: old agent stays connected until the new agent's register call atomically replaces token.
- `POST /api/agent/{id}/rotate-token` on a pending server returns 400.
- `validate_agent_token` skips rows where `token_hash IS NULL` (a pending server's row cannot satisfy any token verification, regardless of how Argon2 verifies the bytes).
- Fingerprint unique index drop migration is applied; two pre-created servers with identical fingerprints both insert successfully.

### Tests removed

All tests under `recovery_merge_*`, `server_recovery_*`, and `recovery_job_*` paths are deleted with their source.

### Manual E2E checklist (added to `tests/`)

1. Add Server with full form → pending card appears with status "Waiting for agent" and code prefix + countdown → run install.sh on test VPS → card flips to online.
2. Recover Agent on an offline server with default revoke_immediately = true → card enters pending → run install.sh → card flips to online.
3. Recover Agent on an online server with revoke_immediately = false → original agent stays online → run install.sh on a different machine → first agent's connection is displaced, new agent becomes online.
4. Delete pending server → row gone, no orphan enrollment rows in DB.
5. Code expiry on pending card → "Regenerate code" produces a fresh plaintext command shown once.
6. Try to Recover on a pending server → action not offered (UI hides it); calling the endpoint directly returns 400.
7. Try to rotate-token on a pending server via API → 400.
