# Agent Registration Redesign

**Status:** Draft
**Author:** ZingerLittleBee
**Date:** 2026-05-25

## Problem

The current agent onboarding flow has two related problems:

1. **"Add Server" doesn't create a server.** Clicking *Add Server* only mints a free-floating enrollment code. The actual `servers` row is created by `POST /api/agent/register` **after** an agent successfully connects with that code. There is no UI representation of "a server I plan to install" — it appears in the list only once the agent comes online.

2. **"Add Server" silently restores existing servers.** The register handler does fingerprint-based dedup: if the new agent's `fingerprint` matches an existing server, it reuses that `server_id` and rotates its token. From the operator's perspective, clicking *Add Server* sometimes creates a brand-new entry and sometimes silently restores an old one, depending on whether the VPS happens to share a fingerprint.

3. **Recover Agent is over-engineered.** The existing `recovery-merge` flow (recovery_job state machine, candidate scoring, rebind WebSocket coordination, recovery_lock service) only works when there is *another online agent* to graft the offline record onto. The normal case — "I reinstalled this VPS, give me a new install command for the same server entry" — is not directly supported and gets shoe-horned into the merge flow.

## Goals

- *Add Server* always creates a new `servers` row, immediately, with all operator-supplied metadata (name, group, tags, billing, remark). The row exists in a `pending` state until an agent connects.
- The enrollment code generated by *Add Server* is bound 1:1 to that pre-created `servers` row. An agent using that code can only bind to that specific row.
- *Recover Agent* opens a dialog similar to *Add Server* but with everything except agent capabilities locked. It produces a new install command bound to the existing server row.
- Fingerprint-based silent restoration is removed entirely. The agent never "discovers" which server it is — the enrollment code is authoritative.
- The complex `recovery-merge` apparatus (state machine, candidate scoring, online-rebind) is removed.

## Non-Goals

- This redesign does not change WebSocket-layer authentication (token + Argon2 verification), agent local capability probing, or the `effective_caps = server_caps & agent_local_caps` rule.
- The `--caps` selection in the dialog continues to be encoded into the `install.sh` command line, not persisted onto the server row. Post-install capability editing remains the job of the existing `capabilities-dialog`.
- iOS / mobile clients are out of scope for this change. They consume `/api/servers` and gain `pending` as a new possible status; they do not initiate registration.

## Architecture

### Status model

A `servers` row is in exactly one of three derived states:

| State | Definition | When it appears |
|---|---|---|
| `pending` | `token_hash IS NULL` | After *Add Server*, before any agent successfully registers. Also after *Recover Agent* if `revoke_immediately = true` and no new agent has connected yet. |
| `online` | `token_hash IS NOT NULL` AND `agent_manager` has a live connection | Normal operation. |
| `offline` | `token_hash IS NOT NULL` AND `agent_manager` has no connection | Agent was online and disconnected, or `revoke_immediately = false` recover where the old agent is still connected (in that subcase, status stays `online` until the new agent takes over). |

The state is not stored as a column; it is derived. This avoids needing to keep a status column in sync with the actual connection state in `agent_manager`.

#### Browser representation

The frontend derives status from two inputs already available:

1. The server DTO returned by `/api/servers` and pushed via `BrowserMessage::FullSync` / `Update` gains a new `has_token: bool` field (= `token_hash IS NOT NULL`). The raw hash and prefix are never sent.
2. The existing `BrowserMessage::ServerOnline { server_id }` / `ServerOffline { server_id }` messages continue to indicate live connection state in `agent_manager`.

Status = `pending` if `!has_token`; otherwise `online` if `ServerOnline` was the last connection event for this id, else `offline`. No new `BrowserMessage` variant is required.

Server-side: when Add Server creates a row, the server is broadcast via the existing `Update` (or `ServerCreated`-style) event already used elsewhere, with `has_token: false`. When Recover with `revoke_immediately = true` clears the token, the same `Update` mechanism re-broadcasts the row with `has_token: false`, plus a `ServerOffline` to drop the live-connection indicator.

### Enrollment binding

`agent_enrollments.target_server_id` is added as a `NOT NULL` foreign key to `servers.id`. Combined with a partial unique index, at most one *unconsumed, unexpired* enrollment can exist per server at a time. This prevents the operator from accidentally minting two install commands for the same server.

```sql
CREATE UNIQUE INDEX idx_enrollments_active_per_server
  ON agent_enrollments(target_server_id)
  WHERE consumed_at IS NULL;
```

### Token revoke semantics in Recover Agent

The Recover dialog has a `Revoke existing token immediately` checkbox (default checked).

- **Checked**: Within the same transaction that mints the enrollment, the server's `token_hash` and `token_prefix` are set to `NULL`, and `agent_manager.remove_connection(server_id)` is called. The old agent is kicked and cannot reconnect (any reconnect attempts will fail with `Unauthorized`). The server immediately enters `pending` state. New agent comes online → `register` sets token → `online`.
- **Unchecked**: Only the enrollment is created. Old token continues to work; old agent stays online. When a new agent calls `/api/agent/register` with the new enrollment, the server-side handler atomically overwrites `token_hash`/`token_prefix` and the WebSocket layer's existing `server_lifecycle_locks` will displace the old connection on its next message attempt.

## Components

### Backend

#### Data model migrations

A single migration adds the binding column and constraint:

```rust
// crates/server/src/migration/m{N}_enrollment_target_server.rs
ALTER TABLE servers
  ALTER COLUMN token_hash    DROP NOT NULL,
  ALTER COLUMN token_prefix  DROP NOT NULL;
-- (SQLite: emulated via table-rebuild pattern used elsewhere in this project.)

DELETE FROM agent_enrollments;  -- enrollments are short-lived (<= 1 day)

ALTER TABLE agent_enrollments
  ADD COLUMN target_server_id TEXT NOT NULL
    REFERENCES servers(id) ON DELETE CASCADE,
  ADD COLUMN revoke_on_mint  INTEGER NOT NULL DEFAULT 0,
  DROP COLUMN label;

CREATE UNIQUE INDEX idx_enrollments_active_per_server
  ON agent_enrollments(target_server_id)
  WHERE consumed_at IS NULL;
```

Per project rules, `up()` is implemented; `down()` is a no-op.

#### API surface changes

| Endpoint | Change |
|---|---|
| `POST /api/agent/register` | No longer creates `servers` rows. Validates enrollment → reads `target_server_id` → sets that row's `token_hash` / `token_prefix` / `last_remote_addr` / `fingerprint` → returns. `fingerprint` is recorded for informational use only; it is not used for lookup or dedup. |
| `POST /api/servers` *(new)* | Admin-only. Creates a `pending` server (token_hash NULL) plus a bound enrollment in one transaction. Request body is the full Add Server form. Response includes `server_id`, `enrollment.code` (one-time), `enrollment.expires_at`. The `max_servers` cap check moves here. |
| `POST /api/servers/{id}/recover` *(new)* | Admin-only. Creates a new bound enrollment for an existing server. Body: `{ revoke_immediately: bool }`. If `revoke_immediately`, atomically NULLs out the server's token columns and disconnects the live agent. Response includes `enrollment.code`, `enrollment.expires_at`. Errors if an active enrollment already exists for this server (caller must delete it first). |
| `POST /api/agent/enrollments` | **Removed.** Replaced by `POST /api/servers` and `POST /api/servers/{id}/recover`. |
| `GET /api/agent/enrollments` | Kept; lists all enrollments with `target_server_id` so the UI can show install command per pending/recovering server. (Alternative: fold into `GET /api/servers` response.) |
| `DELETE /api/agent/enrollments/{id}` | Kept. If the target server is `pending` and has no other enrollment history, the cascading delete drops the server row too. |
| `POST /api/agent/{id}/rotate-token` | Kept. Unchanged. |
| `/api/server-recovery/*` (all variants) | **Removed.** |

#### Register handler simplification

Pseudocode after redesign:

```rust
async fn register(state, addr, headers, body) -> RegisterResponse {
    check_register_rate(ip)?;
    let enrollment = EnrollmentService::verify_and_consume(&state.db, bearer)?
        .ok_or(Unauthorized)?;

    let server = servers::find_by_id(enrollment.target_server_id)?
        .ok_or(Internal("Bound server vanished"))?;  // should not happen given FK + cascade

    let token = AuthService::generate_session_token();
    let hash  = AuthService::hash_password(&token)?;
    let prefix = &token[..8];

    servers::update(server.id)
        .token_hash(hash)
        .token_prefix(prefix)
        .last_remote_addr(ip)
        .fingerprint(body.fingerprint.filter(|f| !f.is_empty()))
        .updated_at(now)
        .exec(&state.db)?;

    audit("agent_enrolled", ...);
    Ok(RegisterResponse { server_id: server.id, token })
}
```

Total LOC drops by ~150 lines vs. current handler.

### Frontend

#### Add Server dialog (redesigned)

The current `add-server-dialog.tsx` collects only `label` + `ttl` + `caps`. The redesign expands it into a richer form:

- **Required**: `name`
- **Optional**:
  - `group_id` (Select, populated from `/api/server-groups`)
  - `tags` (multi-select, populated from `/api/server-tags`)
  - `remark`, `public_remark`
  - Billing block: `price`, `currency`, `billing_cycle`, `billing_start_day`, `expired_at`, `traffic_limit`, `traffic_limit_type`
  - `caps` (checkbox grid, default = CAP_DEFAULT) — affects install.sh `--caps` only
- **Fixed**: TTL = 600s (10 min), displayed as a tip text "Code valid for 10 minutes", no selector
- **On submit**: `POST /api/servers` → dialog switches to "install command" view (existing layout: warning banner + code + install command + 3-step instructions + copy buttons + "Done" / "Add another")
- **List of existing enrollments** is removed from this dialog — pending servers are now represented by `pending`-status cards in the main server list, which is where their install command lives.

#### Pending server card

Server cards with status `pending` render with:

- Status indicator: new color (e.g. amber `bg-amber-500`), distinct from the offline gray and online green dots.
- No metric data; in place of the metric area, show "Waiting for agent…" placeholder + the enrollment code's countdown (`expires in 7m 12s` or `Code expired`).
- Action buttons (in the card menu or footer):
  - **Copy install command**
  - **Regenerate code** — if enrollment expired or was deleted, opens a tiny dialog to mint a new one (uses `POST /api/servers/{id}/recover` with `revoke_immediately = false`, since there is no live token to revoke).
  - **Delete server** — confirmation dialog; cascades enrollment.

#### Recover Agent dialog

A new component `recover-agent-dialog.tsx`, opened from the offline server card's action menu (replacing the old "Recover" entry that used to open `recovery-merge-dialog`).

Layout mirrors Add Server but locked down:

- Server name: read-only text (not an input)
- Caps: checkbox grid, editable (initial value = current `server.capabilities` projected back to keys)
- `Revoke existing token immediately` checkbox, default **checked**
- TTL tip: "Code valid for 10 minutes"
- Submit button: "Generate install command"
- On submit: `POST /api/servers/{id}/recover` → switches to install-command view, same UI as Add Server

If the server already has an active (unconsumed, unexpired) enrollment, the dialog opens directly into the install-command view showing that existing enrollment instead of minting a new one. (The code's plaintext was only revealed at mint time, so this case actually requires showing a "code already issued" notice with the prefix; user must delete the existing enrollment to mint a new one.)

The following frontend files are deleted:

- `apps/web/src/components/server/recovery-merge-dialog.tsx` + test
- `apps/web/src/stores/recovery-jobs-store.ts` + test
- `useRecoveryCandidates`, `startRecoveryMerge` from `apps/web/src/hooks/use-api.ts`
- Corresponding entries from `apps/web/src/lib/api-schema.ts` / `api-types.ts`
- i18n keys: `recovery_merge_*` in `apps/web/src/locales/{en,zh}/servers.json`

### Agent

No agent-side code changes are required. The agent already calls `POST /api/agent/register` with bearer = enrollment code and an optional fingerprint. The server-side behavioral change (binding to a pre-created row instead of creating one) is transparent to the agent. The Welcome message and capability negotiation are unchanged.

## Data Flow

### Add Server (happy path)

```
Operator                Frontend              Server                 DB
   |                       |                     |                     |
   | click Add Server      |                     |                     |
   | fill form, submit ───>|                     |                     |
   |                       | POST /api/servers ─>|                     |
   |                       |                     | BEGIN ─────────────>|
   |                       |                     | INSERT servers      |

   |                       |<─ {server_id, code} |                     |

   |                                              |   Bearer <code>
   |                                              |─────────────────────|
   |                                              | UPDATE servers      |
   |                                              |   SET token_hash=…  |
   |                                              | UPDATE enrollment   |
   |                                              |   SET consumed_at=…|
   |                                              | broadcast ServerOnline
   |                       |<── WS ServerOnline ─|
   |  card flips to online |                     |
```

### Recover Agent (revoke_immediately = true)

```
Operator                Frontend              Server                 DB
   |                       |                     |                     |
   | click Recover         |                     |                     |
   | adjust caps, submit ─>|                     |                     |
   |                       | POST /api/servers/  |                     |
   |                       |   {id}/recover ────>|                     |
   |                       |                     | BEGIN ─────────────>|
   |                       |                     | UPDATE servers      |
   |                       |                     |   SET token_hash=NULL|
   |                       |                     | INSERT enrollment   |

   |                       |                     | broadcast ServerOffline / Pending
   |                       |<─ {code} ───────────|                     |

   |  ...VPS reinstalled, agent registers → online (same as Add Server) |
```

## Errors & Edge Cases

| Case | Behavior |
|---|---|
| Enrollment expired without agent connecting | Pending card shows "Code expired"; operator clicks "Regenerate code" → `POST /api/servers/{id}/recover` with `revoke_immediately = false`. Old expired enrollment row stays for audit until cleanup. |
| Operator clicks Recover twice quickly | Second `POST /api/servers/{id}/recover` fails with `409 Conflict` due to the partial unique index. Frontend handles by refreshing and showing the existing active enrollment. |
| Operator deletes a pending server | Cascade drops the enrollment. No `agent_manager` cleanup needed (there was no connection). |
| Operator deletes an active server | Existing delete flow handles this; the new code does not change `DELETE /api/servers/{id}`. Cascade also handles any active enrollment from a prior Recover. |
| Agent reaches register endpoint with valid code but bound server was deleted | Should not happen: enrollment cascade-deletes with the server. If it does, return `500 Internal`. |
| `max_servers` cap reached | `POST /api/servers` returns `400 BadRequest` before any rows are inserted. The check moves out of the register handler. |
| Two operators race on Add Server | Each transaction inserts its own server row; no conflict. |
| Agent restarts with cached token after `revoke_immediately = true` | Reconnect fails with `Unauthorized` (token_hash is NULL). Agent's existing register-retry path (`MAX_REREGISTER_ATTEMPTS = 3`) does not help because the original enrollment code is consumed. Agent operator must re-run install.sh with the new code. This is acceptable — it's exactly the "I want a fresh agent" intent of `revoke_immediately = true`. |

## Security Considerations

- Enrollment codes remain Argon2-hashed with prefix lookup; binding to a `target_server_id` does not weaken the code itself.
- The partial unique index prevents code stockpiling per server. If the operator wants to invalidate a leaked code, they delete the enrollment row.
- Removing fingerprint dedup eliminates a class of confusion (silent restoration) but also removes a soft "two agents on the same VPS get merged" behavior. The new model requires the operator to explicitly use Recover Agent in that case. This is a deliberate trade-off: explicit > implicit.
- `revoke_immediately = true` is the default in the Recover dialog. Operators must consciously opt out, which keeps secure-by-default semantics.

## Testing

### Unit / integration tests added

- `POST /api/servers` creates one server row + one enrollment row in a single transaction; rolls back on failure.
- `POST /api/servers` respects `max_servers`.
- `POST /api/agent/register` with a bound enrollment updates the bound server's token columns and does **not** create a new server row.
- `POST /api/agent/register` ignores the `fingerprint` field for lookup (records it but does not dedup).
- Partial unique index: a second `POST /api/servers/{id}/recover` while an active enrollment exists returns 409.
- `revoke_immediately = true`: token_hash becomes NULL, `agent_manager.remove_connection` is called, server enters pending state, an `Update` broadcast carries the new `has_token: false` row and a `ServerOffline` event is fired.
- `revoke_immediately = false`: old agent stays connected until new agent's register call atomically replaces token.
- Migration: enrollments table is cleared; existing servers' nullable token columns accept NULL.

### Tests removed

All tests under `recovery_merge_*` and `server_recovery_*` paths are deleted with the source.

### Manual E2E checklist (added to `tests/`)

1. Add Server with full form → pending card appears in list → run install.sh on test VPS → card flips to online.
2. Recover Agent with default revoke_immediately = true on an offline server → card enters pending → run install.sh → card flips to online.
3. Recover Agent with revoke_immediately = false on a running agent → original agent stays online → run install.sh on a different machine → first agent's connection is displaced, new agent becomes online.
4. Delete pending server → row gone, no orphan enrollment.
5. Code expiry on pending card → "Regenerate code" works.

## Open Questions

None at the time of writing. (If `GET /api/agent/enrollments` should be folded into `GET /api/servers` as embedded `pending_enrollment` fields, that's a minor frontend optimization left to the implementation plan.)

## Rollout

This is a single coordinated change touching backend, frontend, and database. There is no incremental rollout; the migration clears the enrollment table (acceptable since enrollments are short-lived). Deploy server and frontend together. Agents do not need to be redeployed.
