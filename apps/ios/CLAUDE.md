# CLAUDE.md — apps/ios

Guidance for Claude Code when working in the **ServerBee iOS client** (`apps/ios`).
This is a native SwiftUI app, not a web port. Reorganize information for an iOS
user — do not mirror the web UI. Follow Apple's Human Interface Guidelines.

## Overview

Native SwiftUI client for the ServerBee server (`crates/server`). It talks to the
same REST + WebSocket API the web SPA uses, via the **mobile auth** endpoints
(`/api/mobile/auth/*`) and a per-installation token pair stored in the Keychain.
iPhone-only, iOS 17+, Swift 6 with `SWIFT_STRICT_CONCURRENCY: complete`.

## Build / Test / Lint

Always `cd apps/ios` first.

```bash
xcodegen generate                         # regenerate the .xcodeproj after ADDING/REMOVING files
xcodebuild -project ServerBee.xcodeproj -scheme ServerBee \
  -destination 'platform=iOS Simulator,id=<SIM_UDID>' \
  -skipPackagePluginValidation build      # build (SwiftLint runs as a build phase)
xcodebuild ... test                       # run the unit test bundle (ServerBeeTests)
```

- **SwiftLint runs as a build-tool plugin**, so `BUILD SUCCEEDED` means lint is
  clean (warnings don't fail the build; treat them as must-fix anyway).
- The project is generated by **XcodeGen** from `project.yml`; sources are
  **globbed** by directory. `.xcodeproj` is gitignored. After adding a new
  `.swift` file you MUST run `xcodegen generate` or it won't be compiled.

## Layout


## JSON coding convention (critical)

`JSONDecoder.snakeCase` / `JSONEncoder.snakeCase` are **plain** coders — they
apply **NO key-conversion strategy and NO date strategy** (kept only for
call-site stability). Therefore:

- **Every** Codable model declares an explicit `enum CodingKeys: String,
  CodingKey` spelling out the wire `snake_case` name for each property
  (e.g. `case cpuUsage = "cpu_usage"`). Forgetting one silently breaks decoding.
  See `Models/README.md`.
- Dates are kept as **RFC3339 `String`** on models (no `Date` decoding strategy).
  Parse for display with the project's tolerant parser; **encode** an outgoing
  RFC3339 timestamp with `WireDate.string(from:)` (a fresh
  `ISO8601DateFormatter` with `.withInternetDateTime`).
- Swift's synthesized `Encodable` omits `nil` optionals (the key is absent),
  which is how request bodies signal "leave unchanged".

## APIClient

`APIClient` is an `actor`. It auto-unwraps the server's `{ "data": T }` envelope.

- `get<T>(path)`, `post<T>(path, body:)`, `postVoid(path, body:)`,
  `put<T>(path, body:)`, `delete<T>(path)`.
- `delete<T>` requires `T: Decodable`. Endpoints that return `{ "data": "ok" }`
  / `{ "data": "revoked" }` must be called as `let _: String = try await
  apiClient.delete(...)` (not `postVoid`/`Void`).
- Errors surface as `APIError.httpError(statusCode: Int, data: Data)`. Map them
  with `AccountSecurityViewModel.message(for: error, fallback:)` (prefers the
  server's `error.message`) or `.errorMessage(from: data)` (takes `Data`).
- Request bodies are encoded with `JSONEncoder.snakeCase`.

### Tri-state for nullable PUT fields

Server PUT endpoints model "nullable + optional" as `Option<Option<T>>`. On the
client use a tri-state encoder (see `UpdateServerRequest.Tri`): `.unchanged`
omits the key, `.clear` sends `null`, `.set(v)` sends the value. Plain (non-
nullable) optional fields just use `encodeIfPresent`.

## ViewModels

- `@MainActor @Observable final class …ViewModel`. Hold `isLoading`/`isSaving`
  and `errorMessage` state; expose `async` methods that `do/catch` around
  `APIClient` and set `errorMessage` via the mapping helpers above.
- Keep network/auth in the view model; views stay declarative.

## WebSocket

- One global `WebSocketClient` (in `ContentView.task`) decodes `BrowserMessage`
  frames and hands them to a `WebSocketRouter`, which fans out to:
  `servers` (ServersViewModel), `alerts`, `security` (SecurityFeedStore),
  `upgrades` (UpgradeJobsStore).
- `full_sync` is routed to BOTH `servers` and `upgrades` (it carries the server
  metrics AND the upgrade snapshot). New message types should decode to
  `BrowserMessage.unknown` rather than throwing, so the receive loop stays quiet.
- Live, push-driven state that several screens read lives in an `@MainActor
  @Observable` **store** injected via `.environment` (e.g. `SecurityFeedStore`,
  `UpgradeJobsStore`) — mirror the existing zustand-store semantics from the web
  when one exists.

## Capability gating

Sections and actions are gated on the server's **effective** capability bitmask
via `CapabilitySet.isEnabled(_:)` (can the agent do it now) and
`.isConfigured(_:)` (is it configured, so historical data stays viewable while
offline). Admin-only UI checks `authManager.user?.role.lowercased() == "admin"`.
Defense-in-depth: the server enforces the same caps/role; the client gate is UX.

## Auth & secrets

- Tokens live in the **Keychain**; never hardcode credentials, tokens, or the
  demo password in source, config, or commits.
- Mobile auth uses a per-installation id (`InstallationID.getOrCreate()`); the
  in-app refresh sends the same id the tokens were minted against.

## SwiftLint limits (from `.swiftlint.yml`)

- `type_body_length` ≤ 250 → move helpers/subviews into a `private extension`
  of the same type (Swift `private`/extension-default members stay callable from
  `body` per SE-0169, since they're same-file).
- Nested type declarations ≤ 1 level.
- `function_body_length` warn 80 / err 150; `file_length` warn 500 / err 1000;
  `line_length` warn 200 / err 300.
- `force_unwrapping` is an opt-in warning — avoid `!`.

## DEBUG visual-verification hooks (`Support/UITestSupport.swift`)

`#if DEBUG` only (compiled out of Release). Driven by launch environment so no
credentials are compiled in. Pass via `simctl launch` using the
`SIMCTL_CHILD_<VAR>` prefix (NOT `--setenv`):

- Auth seed: `SB_UITEST_SERVER` / `SB_UITEST_ACCESS` / `SB_UITEST_REFRESH`
  (+ `_USER_ID` / `_USERNAME` / `_ROLE` / `_INSTALLATION_ID`) → starts signed in.
- `SB_UITEST_TAB` (0=Servers 1=Alerts 2=Insights 3=Settings),
  `SB_UITEST_DEEPLINK` (`server:<id>` / `alert:<key>`),
  `SB_UITEST_SECTION` (`DetailSection.rawValue`),
  `SB_UITEST_PRESENT` (per-view auto-present token, e.g. `edit-server`,
  `upgrade-progress[:stage]`, `insights-maintenance-create`, `advanced-tools`).
- Add a hook when a screen needs a navbar/plain-`Button` tap the headless
  cliclick harness can't reliably trigger, or a backend state the shared demo
  can't produce. See the verification-rig memory for the simulator commands.

## Deferred by design (kept on web)

Full **live** terminal / file-browser / exec interaction (the PTY/streaming
surfaces), GPU metrics, custom-dashboard editing, alert rule/channel
**create-edit** (mobile is read-only + enable toggle + test), plus network-probe
global target config, public status-page config, and IP-quality service
definitions. These show honest entry points / read-only views on mobile.

Note: scheduled-command management (`/api/tasks`, list/create/run/results) and
ping-task config (`/api/ping-tasks`, CRUD) ARE native (Settings → admin) — they
are management surfaces, distinct from the deferred live-exec/terminal PTY.
The temperature history chart is implemented and unit-tested but renders only
when an agent reports temperature (the shared demo has none).
