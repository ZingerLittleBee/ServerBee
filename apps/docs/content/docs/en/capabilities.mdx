ServerBee enforces the principle of least privilege through per-agent capabilities. **Capabilities are owned by the agent host**: each agent computes its own capability set from its config file (and optional CLI flags) and reports it to the server. The server can read and display these capabilities, but it **cannot** change them — there is no server-side toggle.

<Callout type="info">
This is a deliberate trust-boundary decision. The machine that runs the agent is the only place that decides what that agent will do. A compromised or misconfigured server cannot silently turn on a remote shell, file access, or command execution on your hosts.
</Callout>

## Capability List

ServerBee defines 11 capability bits across two risk levels. The valid mask is `2047` (bits 0..=10).

### High Risk (Disabled by Default)

| Capability | Key | Bit Value | Description |
|------------|-----|-----------|-------------|
| **Web Terminal** | `terminal` | `CAP_TERMINAL` (1) | Allow opening a remote terminal via browser |
| **Remote Exec** | `exec` | `CAP_EXEC` (2) | Allow remote command execution |
| **File Manager** | `file` | `CAP_FILE` (64) | Allow remote file browsing, editing, upload/download |
| **Docker Management** | `docker` | `CAP_DOCKER` (128) | Allow Docker container monitoring, log streaming, and container actions |

<Callout type="warn">
These capabilities allow executing arbitrary code or accessing the filesystem on the target server, so they are disabled by default. Enable them only on trusted hosts by editing that host's agent config.
</Callout>

<Callout type="info">
File Manager requires additional agent-side configuration (`root_paths`, `deny_patterns`) for path sandbox security. See the [Agent Setup](/en/docs/agent) and [Configuration](/en/docs/configuration) pages for details.
</Callout>

### Low Risk (Enabled by Default)

| Capability | Key | Bit Value | Description |
|------------|-----|-----------|-------------|
| **Auto Upgrade** | `upgrade` | `CAP_UPGRADE` (4) | Allow remote binary upgrades |
| **ICMP Ping** | `ping_icmp` | `CAP_PING_ICMP` (8) | Allow ICMP probe tasks |
| **TCP Probe** | `ping_tcp` | `CAP_PING_TCP` (16) | Allow TCP port probe tasks |
| **HTTP Probe** | `ping_http` | `CAP_PING_HTTP` (32) | Allow HTTP probe tasks |
| **Security Events** | `security_events` | `CAP_SECURITY_EVENTS` (256) | Allow agent to report SSH login / brute-force / port-scan events (see [Security Events](/en/docs/security-events)) |
| **Firewall Blocklist** | `firewall_block` | `CAP_FIREWALL_BLOCK` (512) | Allow agent to apply the server-pushed nftables blocklist. Requires root or `CAP_NET_ADMIN` plus the `nft` CLI on the host. See [Firewall Blocklist](/en/docs/firewall) |
| **IP Quality** | `ip_quality` | `CAP_IP_QUALITY` (1024) | Allow agent to query third-party IP quality APIs for outbound IP scoring |

An agent with no `[capabilities]` overrides defaults to `1852` (auto upgrade + the three ping probes + security events + firewall blocklist + IP quality), which keeps the high-risk terminal, exec, file, and Docker capabilities off.

## Configuring Capabilities (on the agent host)

An agent computes its capability bitmask as:

1. Start from the built-in default set (`CAP_DEFAULT = 1852`).
2. Apply the config file `[capabilities]` block — `allow` adds bits, `deny` removes bits. Within this layer, `deny` wins over `allow`.
3. Apply CLI `--allow-cap` / `--deny-cap` flags on top, for ad-hoc overrides. The CLI layer wins over the config layer.

Unknown capability keys are a hard error at startup, so a typo fails fast rather than silently dropping a capability.

### Config file

Edit `/etc/serverbee/agent.toml` (or `agent.toml` in the working directory) on the agent host:

```toml
[capabilities]
# Turn on high-risk capabilities this host should expose.
allow = ["terminal", "file"]
# Strip default capabilities this host should not expose.
deny = ["ip_quality"]
```

Restart the agent for changes to take effect:

```bash
systemctl restart serverbee-agent
```

### Environment variables

The same block can be supplied via environment variables (Figment, `SERVERBEE_` prefix, `__` nesting):

```bash
SERVERBEE_CAPABILITIES__ALLOW=["terminal","file"]
SERVERBEE_CAPABILITIES__DENY=["ip_quality"]
```

### CLI flags

For one-off overrides, pass repeated flags when launching the agent:

```bash
serverbee-agent --allow-cap terminal --allow-cap file --deny-cap ip_quality
```

### At install time

The install script accepts `--caps` to seed the agent's capability config during enrollment:

```bash
curl -fsSL https://your-server/install.sh | sh -s -- --caps terminal,file
```

Running the installer interactively also prompts for capabilities, with the defaults pre-checked.

## Enforcement model

The agent host is the single source of truth, but enforcement happens at two boundaries for defense in depth.

### Server-side gating (based on reported capabilities)

The server records what each agent reports and rejects control-plane requests for capabilities the agent has not enabled:

- **Terminal**: the WebSocket upgrade is rejected with 403
- **Exec**: `POST /api/tasks` and scheduled task runs filter out servers whose agent lacks `exec` and write synthetic results (`exit_code = -2`)
- **Auto Upgrade**: `POST /api/servers/{id}/upgrade` returns 403 when `upgrade` is not reported
- **Ping and Traceroute**: probe tasks are filtered by capability; traceroute requires `ping_icmp`
- **File Manager**: file endpoints reject requests before dispatch when `file` is not reported
- **Docker**: Docker read/action endpoints and the Docker log WebSocket routes require `docker` and agent runtime Docker support

Because capabilities are agent-owned, the denial reason is always `agent_capability_disabled`.

### Agent-side enforcement

Even if a server-side request slips through, the agent re-checks capabilities locally:

- It returns a `CapabilityDenied` message for unauthorized commands
- On receiving `CapabilityDenied`, the server writes a synthetic result (`exit_code = -1`)
- Denial events are recorded in the audit log

### Reporting and display

When an agent connects (or reconnects), it sends `SystemInfo` carrying `agent_local_capabilities`. The server:

1. Persists that value into the `servers.capabilities` column as a display mirror (so the dashboard can show capabilities even while the agent is offline).
2. Broadcasts `CapabilitiesChanged` to all connected browsers so the UI reflects the current set in real time.

If ping-related bits change between reconnects, the server automatically re-syncs ping tasks for the agent.

## Frontend behavior

The web and iOS clients display capabilities **read-only**:

- **Server Detail → Capabilities**: a read-only summary of which capabilities the agent has enabled, with a note that they are configured in the agent config file
- **Settings → Capabilities**: a read-only fleet matrix showing the enabled/disabled state per server
- **Tasks page**: servers whose agent lacks `exec` are greyed out
- **Terminal button**: hidden for servers whose agent lacks `terminal`
- **Files button**: hidden for servers whose agent lacks `file`
- **Docker link**: hidden for servers whose agent lacks `docker`

## Runtime capability fields

The server response exposes three capability fields, which are all equal under the agent-owned model:

- `capabilities`: the persisted mirror of the last value the agent reported
- `agent_local_capabilities`: the live value reported by the connected agent
- `effective_capabilities`: the value actually enforced (identical to the agent-local value)

The fields are kept distinct in the API for forward compatibility and so an offline server can still display its last-known capabilities from the mirror.
