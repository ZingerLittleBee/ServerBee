ServerBee Agent 会在每台主机上检测三类主机级安全事件，并将其推送到 Server 端供查询、筛选和告警触发。检测完全在 Agent 上完成，主机日志不会离开本机，只有结构化事件元数据被上报。

## 事件类型

| 类型 | 检测内容 | 严重度 |
|------|----------|--------|
| `ssh_login` | 一次成功的 SSH 登录。如果 `(user, source IP)` 组合从未出现过，则标记 `first_seen=true` | `info` |
| `ssh_brute_force` | 同一源 IP 在滑动窗口内累计大量 SSH 失败尝试 | `medium` / `high` / `critical`（按尝试的不同用户名数量升级）|
| `port_scan` | 同一源 IP 在滑动窗口内命中大量不同的目标端口 | `medium` |

每个事件都附带结构化证据（失败次数、用户名集合、扫描端口列表等）以及检测来源（`journal`、`auth_log`、`conntrack`、`nflog`）。

## 前置条件

- **仅支持 Linux。** SSH 检测依赖 systemd journal 或 `/var/log/auth.log`，端口扫描依赖 `conntrack`。
- 需要在 Server 端启用 **`CAP_SECURITY_EVENTS` capability**（位值 `256`，新 Agent 默认启用）。
- 端口扫描检测需要主机安装 `conntrack` CLI（`apt install conntrack` / `dnf install conntrack-tools`），并将 `security.port_scan.enabled` 设为 `true`。

## 查看事件

- **总览页**：`/security` 列出所有服务器的事件，包含 24 小时分类统计 KPI、7 天时间线和支持事件类型 / 严重度 / 源 IP / 时间范围筛选的表格。
- **单机视图**：服务器详情页的 **Security** Tab 显示当前服务器范围内的同款视图。
- **事件详情**：点击表格行打开抽屉，展示完整证据、检测来源以及源 IP 的 GeoIP 信息（若已配置 GeoIP）。
- **实时推送**：新事件通过 WebSocket 即时显示，无需刷新。

## 检测器调优

阈值在 Agent 上配置，按主机生效。默认值偏保守，避免在繁忙跳板机上产生噪音。

| TOML 键 | 环境变量 | 默认值 | 说明 |
|---------|----------|--------|------|
| `security.enabled` | `SERVERBEE_SECURITY__ENABLED` | `true` | 安全检测器总开关 |
| `security.ssh.window_seconds` | `SERVERBEE_SECURITY__SSH__WINDOW_SECONDS` | `60` | SSH 爆破检测滑动窗口长度（秒）|
| `security.ssh.failed_threshold` | `SERVERBEE_SECURITY__SSH__FAILED_THRESHOLD` | `10` | 窗口内累计失败达到该值即触发一个 `ssh_brute_force` 事件，触发后队列清空 |
| `security.port_scan.enabled` | `SERVERBEE_SECURITY__PORT_SCAN__ENABLED` | `false` | 端口扫描检测开关。需要 `conntrack` CLI |
| `security.port_scan.window_seconds` | `SERVERBEE_SECURITY__PORT_SCAN__WINDOW_SECONDS` | `30` | 端口扫描滑动窗口长度（秒）|
| `security.port_scan.distinct_port_threshold` | `SERVERBEE_SECURITY__PORT_SCAN__DISTINCT_PORT_THRESHOLD` | `20` | 同一源 IP 在窗口内命中的不同端口数达到该值即触发一个 `port_scan` 事件 |
| `security.data_dir` | `SERVERBEE_SECURITY__DATA_DIR` | `/var/lib/serverbee/security` | 持久化 `first_seen` 存储目录，记录已知 `(user, IP)` 组合 |

完整环境变量参考见 [配置文档](/cn/docs/configuration)。

<Callout type="info">
sshd 对一次失败尝试通常会写两行日志（`Invalid user ...` + `Failed password ...`）。在默认 `failed_threshold=10` 下，同一 IP 每发起约 5 次真实失败尝试，就会产生一个 `ssh_brute_force` 事件。
</Callout>

## 严重度升级（SSH 爆破）

`ssh_brute_force` 的严重度反映攻击者在窗口内尝试过的不同用户名数量：

| 不同用户名数 | 严重度 |
|--------------|--------|
| 1 | `medium` |
| 2 – 4 | `high` |
| ≥ 5 | `critical`（撞库）|

事件证据还会附带 `invalid_user_count`（无效用户失败次数）和 `sample_users`（前 5 个不同用户名样本），便于快速识别常见扫库特征。

## 配置告警

**Settings → Alerts** 支持三种事件驱动的规则类型：

| 规则类型 | 触发条件 |
|----------|----------|
| `ssh_login_detected` | 任意 `ssh_login` 事件（可选仅在 `first_seen=true` 时触发）|
| `ssh_brute_force_detected` | 严重度不低于配置阈值的 `ssh_brute_force` 事件 |
| `port_scan_detected` | 任意 `port_scan` 事件 |

### 快速创建

Alerts 页面提供三张 **预设卡片**，一键创建规则。预设已填入合理默认值，只需选择通知组并（可选）指定生效的服务器范围。

### 过滤参数

每条安全规则支持：

- **`severity_min`** — 仅在不低于指定严重度时触发（例如爆破规则可设为 `high`）
- **`exclude_cidrs`** — 屏蔽来自可信网段的事件（例如 `["10.0.0.0/8", "192.168.0.0/16"]`）
- **`first_seen_only`**（仅 `ssh_login`）— 仅在出现新 `(user, IP)` 组合时通知

### 去重

通知按 `(rule_id, server_id, event_key)` 维度去重。`event_key` 包含源 IP，因此两个不同的攻击者命中同一台服务器仍会产生两次通知；而同一攻击者在去重窗口内反复触发则只产生一次通知。

<Callout type="warn">
安全规则类型不能与指标规则混在同一条 alert rule 中，且每条规则只允许包含一个安全 item。校验器会强制这一约束 —— 不同事件类型请分别建独立规则。
</Callout>

## 数据保留

安全事件默认保留 **30 天**。可通过 `retention.security_event_days`（环境变量 `SERVERBEE_RETENTION__SECURITY_EVENT_DAYS`）调整。清理任务每天自动清除过期数据。

## 数据链路

```
sshd / 内核
  ↓ (journal / auth.log / conntrack)
Agent 检测器
  ↓ (AgentMessage::SecurityEvent，WebSocket)
Server：security_event 表 + 告警评估 + 浏览器广播
  ↓
浏览器：/security 页面 + Security Tab + WebSocket 推送
  ↓
通知组（Email / Webhook / Telegram / ...）
```
