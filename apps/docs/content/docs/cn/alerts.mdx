ServerBee 提供灵活的告警系统，支持多种指标类型的阈值监控、多种通知渠道以及精细的触发控制。

## 告警概述

告警系统的工作流程：

```
后台任务 (每 1 分钟)
  --> 遍历所有启用的告警规则
  --> 对每条规则的覆盖范围内的服务器逐一评估
  --> 满足触发条件 --> 通过通知组发送通知
  --> 从触发状态恢复 --> 发送恢复通知（如配置）
```

## 创建告警规则

每条告警规则包含以下核心要素：

- **规则名称**：用于标识和展示
- **告警条件**：一条或多条指标阈值条件（AND 关系）
- **覆盖范围**：适用的服务器范围
- **触发模式**：always（持续通知）或 once（仅首次通知）
- **通知组**：告警触发后的通知方式
- **关联任务**：触发/恢复时自动执行的远程命令（可选）

## 支持的指标类型

ServerBee 支持 14 种以上的告警指标：

### 资源阈值类

| 指标类型 | 判定依据 | 示例 |
|----------|----------|------|
| `cpu` | CPU 使用率 (%) | CPU > 90% |
| `memory` | 内存使用率 (%) | 内存 > 85% |
| `swap` | Swap 使用率 (%) | Swap > 50% |
| `disk` | 磁盘使用率 (%) | 磁盘 > 90% |
| `load1` | 1 分钟负载 | load1 > 10 |
| `load5` | 5 分钟负载 | load5 > 8 |
| `load15` | 15 分钟负载 | load15 > 5 |
| `temperature` | 传感器温度 (C) | 温度 > 80C |
| `gpu` | GPU 平均使用率 (%) | GPU > 95% |
| `tcp_conn` | TCP 连接数 | TCP > 10000 |
| `udp_conn` | UDP 连接数 | UDP > 5000 |
| `process` | 进程数 | 进程 > 500 |
| `net_in_speed` | 入站网络速率 (bytes/s) | 入站 > 100MB/s |
| `net_out_speed` | 出站网络速率 (bytes/s) | 出站 > 100MB/s |

### 流量周期类

用于监控一段时间内的累计流量：

| 指标类型 | 说明 |
|----------|------|
| `transfer_in_cycle` | 周期内入站累计流量 |
| `transfer_out_cycle` | 周期内出站累计流量 |
| `transfer_all_cycle` | 周期内双向累计流量 |

周期选项：`hour` / `day` / `week` / `month` / `year`

### 网络质量

| 指标类型 | 说明 |
|----------|------|
| `network_latency` | 平均探测延迟超过阈值时触发 |
| `network_packet_loss` | 丢包率超过阈值时触发 |

### 离线检测

| 指标类型 | 说明 |
|----------|------|
| `offline` | 服务器持续离线超过指定时长后触发 |

## 阈值配置

每个告警条件支持设置最小值和最大值：

```json
{
  "rule_type": "cpu",
  "min": null,
  "max": 90.0
}
```

- **max**：当指标值超过此阈值时触发（上限告警）
- **min**：当指标值低于此阈值时触发（下限告警）
- 可以同时设置 min 和 max，形成范围告警

一条告警规则可以包含多个条件，所有条件必须同时满足（AND 逻辑）才会触发告警。例如：

```json
[
  { "rule_type": "cpu", "max": 90.0 },
  { "rule_type": "memory", "max": 85.0 }
]
```

上述规则表示：CPU 超过 90% 且内存超过 85% 时才触发。

## 覆盖类型

告警规则的覆盖范围有三种模式：

| 覆盖类型 | 说明 |
|----------|------|
| `all` | 适用于所有服务器 |
| `include` | 仅适用于指定的服务器列表 |
| `exclude` | 适用于除指定服务器外的所有服务器 |

## 告警评估机制

告警评估并非简单的单点阈值判断，而是采用了采样窗口和触发比例的机制，有效降低因瞬时波动导致的误报。

### 采样窗口

- 查询最近 **10 分钟** 的指标记录（即 10 个采样点，因为每分钟写入一次）
- 检查其中超过阈值的采样点比例

### 触发比例

- **70%** 以上的采样点超过阈值才判定为触发
- 即 10 个采样点中至少 7 个超过阈值

### 通知去抖

- `always` 模式下，同一（规则 + 服务器）组合的最短通知间隔为 **5 分钟**
- 避免在持续异常时频繁发送通知

### 触发模式

| 模式 | 行为 |
|------|------|
| `always` | 每次评估满足条件都发送通知（受 5 分钟去抖限制） |
| `once` | 仅在首次触发时通知，恢复后再次触发才会再次通知 |

### 状态持久化

告警触发状态会持久化到 SQLite 数据库的 `alert_states` 表中。这意味着：

- Server 重启后，`once` 模式的规则不会重复触发
- 已触发未恢复的告警状态在重启后自动恢复
- 启动时从数据库加载热缓存，加速后续评估

### 恢复机制

当一条之前触发的告警规则不再满足触发条件时：

1. 标记为已恢复
2. 如果配置了恢复触发任务 (`recover_trigger_tasks`)，自动执行对应的远程命令
3. 清除告警状态，下次满足条件时可以重新触发

## 通知渠道

ServerBee 支持以下通知渠道：

### Webhook

通过 HTTP 请求发送通知，适用于集成到各类自动化平台。

| 配置项 | 说明 |
|--------|------|
| `url` | Webhook 目标 URL |
| `method` | HTTP 方法（GET / POST） |
| `headers` | 自定义请求头 |
| `body_template` | 请求体模板 |

### Telegram

通过 Telegram Bot 发送消息。

| 配置项 | 说明 |
|--------|------|
| `bot_token` | Telegram Bot Token |
| `chat_id` | 目标聊天 ID |

### Bark

通过 Bark 推送 iOS 通知。

| 配置项 | 说明 |
|--------|------|
| `server_url` | Bark 服务端地址 |
| `device_key` | 设备 Key |

### Email

通过 SMTP 发送邮件通知。

| 配置项 | 说明 |
|--------|------|
| `smtp_host` | SMTP 服务器地址 |
| `port` | SMTP 端口 |
| `username` | SMTP 用户名 |
| `password` | SMTP 密码 |
| `from` | 发件人地址 |
| `to` | 收件人地址 |

### 模板变量

通知内容支持以下模板变量：

| 变量 | 说明 |
|------|------|
| `{{server_name}}` | 服务器名称 |
| `{{server_id}}` | 服务器 ID |
| `{{rule_name}}` | 告警规则名称 |
| `{{event}}` | 事件类型（触发/恢复） |
| `{{message}}` | 告警详细信息 |
| `{{time}}` | 事件发生时间 |
| `{{cpu}}` | 当前 CPU 使用率 |
| `{{memory}}` | 当前内存使用率 |

默认通知模板：

```
[ServerBee] {{server_name}} {{event}}
{{message}}
时间: {{time}}
```

### 发送测试

创建通知渠道后，可以通过测试接口验证配置是否正确：

```
POST /api/notifications/:id/test
```

## 通知组

通知组用于将多个通知渠道组合在一起。一条告警规则关联一个通知组，触发时会同时通过组内所有渠道发送通知。

例如，你可以创建一个名为「紧急告警」的通知组，同时包含 Telegram 和 Email 两个渠道，确保重要告警不会遗漏。

## 离线检测告警

离线检测是一种特殊的告警类型，用于监控服务器的在线状态：

- 通过后台任务每 10 秒扫描 Agent 连接状态
- 当 Agent 最后一次上报时间超过 30 秒时判定为离线
- 配合 `offline` 类型的告警规则，可以在服务器离线指定时长后发送通知

```json
{
  "rule_type": "offline",
  "duration": 60
}
```

上述规则表示：服务器持续离线超过 60 秒后触发告警。
